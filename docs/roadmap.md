# 路线图

路线图会随着实际使用反馈调整，以下是按优先级粗略整理的方向。

---

## 待办清单（按优先级排序）

### 🔥 优先实现

- [ ] **Enable 开关即时生效**
  - 复杂度：低
  - 问题：当前 enable 开关只更新草稿状态，需点击"应用修改"才生效
  - 方案：新增 `toggleRuleEnabled` 方法，enable 开关直接同步更新 `appliedRules` 和 `mockRules`，其他配置仍保持"应用修改"工作流

- [ ] **API 列表筛选与视觉优化**
  - 复杂度：低
  - 问题：API 数量多时难以快速定位已启用的 API，小圆点指示器不够醒目
  - 方案：
    - 新增筛选控件（全部 / 已启用 / 已禁用）
    - 改进视觉样式：启用的 API 添加绿色左边框 + 淡绿色背景 + "ON" 徽章

- [ ] **规则导入/导出**
  - 复杂度：低
  - 方案：JSON 序列化/反序列化，支持文件下载和上传

- [ ] **Field Omission 的 UI 配置入口**
  - 复杂度：中
  - 说明：core 已支持，UI 尚未开放

### 📋 需要设计

- [ ] **API 级响应预设**
  - 复杂度：中
  - 当前业务模板是全局的，所有 API 共享
  - 支持为单个 API 保存多套响应预设（如：正常、空数据、分页边界、特定异常）
  - 快速切换当前 API 生效的预设

- [ ] **批量编辑与批量应用**
  - 复杂度：中
  - 支持多选 API，批量启用/禁用/配置

- [ ] **更完善的规则匹配能力**
  - 复杂度：中高
  - 支持 query/body/header 维度匹配（在不破坏现有行为的前提下逐步扩展）

- [ ] **API 解析增强**
  - 复杂度：中
  - 支持更多组织方式（例如多文件、导入 URL 常量、简单拼接等）

### ⏳ 暂缓

- [ ] **基于 Response TS 类型的 mock 模板生成**
  - 复杂度：高
  - 从 TS 类型构建 schema（或基于 JSDoc/示例）生成初始 `result`
  - UI 一键生成/刷新，并允许手工编辑
  - 需要在构建期做类型解析（`tsconfig`/类型展开/跨文件引用）

- [ ] **更丰富的网络模拟**
  - 复杂度：中
  - 带宽/分片、重试抖动、阶段性错误等

- [ ] **可插拔的 mock 数据生成器**
  - 复杂度：中
  - 集成 faker/seed 等库

- [ ] **团队共享规则**
  - 复杂度：高
  - 需要后端服务支持（远程存储/同步）
